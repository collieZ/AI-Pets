# AI-Pets 硬件产品方向

硬件产品方向关注把 AI-Pets 从桌面窗口扩展到实体设备。核心目标是用受控渲染、传感器和 LLM 连接形成可长期运行的陪伴式宠物体验。

## 显示/运行架构

硬件端可以采用主控设备加显示模块的架构。显示层负责播放宠物状态，运行层负责协议解析、事件调度、网络连接和本地缓存。

可选形态包括：

- 小尺寸 LCD 或 OLED 屏幕。
- Linux SBC 加轻量渲染进程。
- MCU 加外部存储和简化 spritesheet 播放。
- 手机或桌面端作为配置工具。

硬件端应继续读取 AI Pet Protocol manifest，但可以只支持协议子集。

## 受限渲染器

受限渲染器需要明确能力边界。硬件端优先支持 spritesheet atlas、固定帧率播放、循环状态和一次性状态。复杂透明效果、滤镜、骨骼动画和大尺寸资源可以不进入首版。

建议约束：

- 限制 atlas 尺寸和帧数。
- 限制同时加载的宠物数量。
- 预先缩放或转码资源。
- 只支持 manifest 中声明的能力子集。
- 在不支持气泡文本的设备上改用短文本页或灯效提示。

## 传感器

传感器可以让宠物响应真实环境。首批可验证输入包括：

- 触摸或按键。
- 加速度计，用于拿起、摇晃和倾斜。
- 麦克风音量事件，用于感知呼唤或环境变化。
- 光照传感器，用于日夜状态。
- 距离传感器，用于靠近和离开。

传感器事件应转换为协议层 interaction 或语义事件。例如触摸可以触发 `click`，长时间无操作可以触发 `idle`，靠近可以触发 `greet`。

## LLM 连接

硬件端可以通过云端或本地代理连接 LLM。设备不一定直接运行大模型，但需要有稳定的事件协议。

推荐分层：

- 设备端负责显示、传感器和短期状态。
- 网关或手机端负责鉴权、网络和模型请求。
- LLM 返回文本、情绪或任务状态。
- 事件桥接把模型状态映射为 `aiThinking`、`aiWorking`、`aiDone`、`aiError` 等 interaction。

为了保护体验，LLM 延迟期间应有等待或思考状态，失败时应有明确错误状态和可恢复路径。

## OTA/内容更新

硬件产品需要支持固件、渲染器和宠物内容更新。OTA 与内容更新应分层处理：

- 固件 OTA 更新系统能力和安全修复。
- 渲染器更新协议支持和播放能力。
- 内容更新下发宠物包、状态、气泡文本和活动主题。
- 更新前校验 manifest、资源大小和协议版本。
- 更新失败时保留上一版可用内容。

宠物内容应支持版本号、签名校验和灰度发布。硬件端下载新包后先校验，再切换到新内容，避免设备进入不可显示状态。
